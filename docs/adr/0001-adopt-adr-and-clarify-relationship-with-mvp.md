# 1. ADR を採用し MVP との役割分担を明確にする

Date: 2026-06-23

## Status

Accepted

## Context

これまで設計・方針に関する意思決定は `docs/mvp.md` に追記していくスタイルを取っていた。このスタイルには次の課題がある。

- 1つのファイルに共有式で追記し続けると、MVP が肥大化し読み返しづらくなる。
- 実装者が並列に作業するようになると、同一ファイルへの追記が衝突（コンフリクト）を起こしやすい。

意思決定ログを残すだけならコミットメッセージでも可能だが、後から読み返す際にコンテキストを辿りにくい。

ADR の仕様では、基本的に最新を見ればよく、必要なコンテキストがある場合のみ過去の関連 ADR を辿ればよい。この性質により、ログが単なる追記の羅列になって読み返しが大変になる、という懸念は緩和される。

つまり「MVP に方針を追記していくスタイル」は実質的に ADR が担うべき役割であり、ADR を正式に導入したうえで MVP と ADR の役割を分ける必要がある。

## Decision

ADR を意思決定の記録として採用する

そのうえで MVP と ADR の役割を次のように分ける。

- **MVP（`docs/mvp.md`）はスコープを書く。** 何を最小要件として作るかを定義する。
- **ADR はその判断を書く。** なぜその方針・選択をしたのかという意思決定を記録する。

スコープと判断の両方に影響する変更では、MVP と ADR の両方を更新する場合がある。

## Consequences

- 意思決定が MVP から分離され、MVP が肥大化しにくくなる。並列実装時のコンフリクトも起こりにくくなる。
- 意思決定は ADR として MVP 完成後も長期的に参照できる。
- 変更時には「スコープの変更か（MVP）」「判断の記録か（ADR）」「両方か」を判断する必要がある。運用に慣れるまでは判断の指針をルール化していく。
